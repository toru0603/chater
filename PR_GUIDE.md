# PR ガイドライン

このファイルはプルリクエスト（PR）作成時の最低限のガイドラインとテンプレートを示します。

## タイトルの例

- `feat: 新しい機能の簡潔な説明`
- `fix: 不具合の簡潔な説明`
- `docs: ドキュメント更新`

## 説明テンプレート

```
概要:
- 何を実装/修正したか（短く）

動機:
- なぜこの変更が必要か

変更点:
- 主な変更点を箇条書き

動作確認手順:
- ローカルでの確認手順やコマンド

テスト:
- 実行したテストと結果

関連 Issue:
- Fixes #<番号>
```

## チェックリスト（PR で確認すること）

- [ ] テストを追加/更新した
- [ ] すべてのテストが通る
- [ ] ドキュメントを更新した（必要な場合）
- [ ] レビュアーをアサインした
- [ ] CI が成功している

## マージポリシー

- 原則として、CI が成功し、レビュアーの承認を 1 件以上得てからマージします。
- 履歴を整理するために `Squash and merge` を推奨します（プロジェクト方針に応じて変更可）。

## レビュー時の注目ポイント

- 変更が仕様に合っているか
- テストが十分か（境界ケースを含める）
- セキュリティ上の懸念がないか
- 後方互換性が守られているか

## ローカルでのテスト実行

```bash
pip install -r requirements.txt
pytest -q
```

不明点や大きな設計変更は事前に Issue を作成して議論してください。
