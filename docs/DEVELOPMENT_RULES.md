# 開発ルール

このドキュメントは開発チームが従うべきルールをまとめます。README をドキュメントの入口とし、すべての Markdown が README から辿れることを前提に運用してください。

## 概要

- 目的: 開発フロー、コミット・PR の慣習、CI 運用などを明文化して一貫性を保つこと。

## ブランチと作業フロー

- ブランチ命名: `feature/<説明>`, `fix/<説明>`, `docs/<説明>` を基本とします。
- ブランチは必ず `origin/main` から作成してください。例:
  - `git fetch origin`
  - `git checkout -b feature/your-short-desc origin/main`
- 作業中は main を最新に保ち、PR 作成前に `git rebase origin/main` を行ってください。

## コミット

- コミットメッセージは Conventional Commits スタイルを推奨します（例: `feat: ~`, `fix: ~`, `docs: ~`）。
- 1 行目に要約（50 文字以内）、空行、必要なら詳細説明を追加してください。

## プルリクエスト

- PR 本文に以下を含めること: 概要、動機、主要変更点、動作確認手順、テスト内容。
- PR チェックリスト（最低）:
  - テストを追加/更新した
  - すべてのテストが通る
  - 必要なドキュメントを更新した（README の導線含む）
  - CI が成功している
  - レビュワーを 1 人以上アサインした
- マージポリシー: CI 合格かつレビュー承認後にマージ。Squash and merge を推奨します。

## テストと CI

- 変更にはユニットテスト／統合テストを追加してください。
- ローカル実行: `pytest -q` を実行し、テストが通ることを確認してください。
- リポジトリには Unit / E2E / Typecheck 等の必須ステータスチェックがあります。CI が成功するまでマージしないでください。

## コード品質

- PEP8 準拠を基本とし、`black` 等の自動整形ツールの利用を推奨します。
- 型チェック（`mypy`）を導入している場合は通すこと。

## Copilot と自動補助

- Copilot 自動適用の方針は `.github/COPILOT_REVIEW.md` を参照してください。

## ドキュメント運用

- 新しいドキュメントを追加・移動・改名した場合は、README のドキュメント一覧にリンクを追加してください（CONTRIBUTING.md にもルールを明記しています）。

## その他

- 破壊的な API 変更や大きな設計変更は事前に Issue を立て、関係者と合意を得てから実装を進めてください。
